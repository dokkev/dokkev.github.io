# WBC02. Representing Floating-Base Robots in Code

In the previous post, we looked at Whole-Body Control from the system level.

The main point was that WBC is not just a torque calculator. It is a control architecture for floating-base robots that must coordinate tasks, contacts, balance, constraints, and actuator limits.

Before we can build that architecture, we need to answer a more basic question:

> How should a floating-base robot be represented in code?

This sounds like an implementation detail.

It is not.

Many WBC bugs start from a wrong mental model of the robot state. For a fixed-base manipulator, it is often safe to think in terms of joint position, joint velocity, and joint torque:

```cpp
q     // joint position
qdot  // joint velocity
tau   // joint torque
```

For an 8-DoF robot hand, this is usually fine.

```cpp
nq = 8
nv = 8
na = 8
```

The configuration dimension, velocity dimension, and actuator dimension are all the same. The code can store `q`, `qdot`, and `tau` using the same joint ordering.

But once a floating base enters the system, this simple picture breaks.

A floating-base robot does not only have actuated joints. It also has a base pose and base velocity. That base can move in space, but it is not directly actuated.

This is where the state representation becomes more subtle.

---

## 1. Fixed-base versus floating-base robots

A fixed-base system has a base rigidly attached to the world.

A typical industrial manipulator bolted to a table is a fixed-base robot. A robot hand attached to a wrist can also be treated as fixed-base if the wrist frame is considered fixed for the purpose of hand control.

In this case, the robot motion is described only by the actuated joints:

```cpp
q = [joint positions]
v = [joint velocities]
```

The generalized coordinates are basically the joint coordinates. The generalized velocity is basically joint velocity. The actuator torque vector has the same dimension.

This is why fixed-base robot code can often get away with a simple state structure:

```cpp
struct RobotState {
  Vector q;
  Vector qdot;
  Vector tau;
};
```

For a floating-base robot, the base itself moves.

Humanoids, quadrupeds, free-floating manipulators, and floating hand-arm systems fall into this category. The robot state must include both the base motion and the actuated joint motion.

Conceptually:

```cpp
q = [base pose, joint positions]
v = [base velocity, joint velocities]
```

The floating base has 6-DoF rigid body motion:

```text
translation: 3 DoF
rotation:    3 DoF
total:       6 DoF
```

So compared to a fixed-base robot, a floating-base robot has six additional generalized velocity components.

However, these six base degrees of freedom are not directly actuated.

There is no motor that applies torque directly to the floating base in world coordinates. The motors are located at the joints. The base moves because of joint torques, contact forces, gravity, and inertia.

This is why floating-base robots are generally underactuated.

For example, consider a floating-base system with 8 actuated joints.

The generalized velocity dimension is:

```cpp
nv = 6 + 8 = 14
```

But the actuator dimension is only:

```cpp
na = 8
```

The dynamics are solved in a 14-dimensional generalized velocity space, but the actual actuator command has only 8 dimensions.

This is the reason selection matrices appear in floating-base dynamics:

```text
M(q) qddot + h(q, v) = Sᵀ tau + Jᵀ lambda
```

Here, `Sᵀ tau` injects actuator torque only into the actuated joint equations. The floating-base part of the dynamics receives no direct actuator torque.

Conceptually:

```text
Sᵀ tau = [0_base_6, tau_actuated]
```

This is a small notation detail in a paper, but it is a major design constraint in code.

The controller must never confuse generalized dynamics space with actuator command space.

---

## 2. Why is a humanoid underactuated if every joint has an actuator?

At this point, a natural question comes up.

> Humanoids usually have actuators at almost every joint.
> So why do we call them underactuated?

The answer is that the issue is not whether the mechanism has many actuators.

The issue is whether every generalized degree of freedom in the dynamics can be directly actuated.

A humanoid may have actuators at the hip, knee, ankle, shoulder, elbow, wrist, and neck. But the floating base itself is not directly actuated. There is no motor between the robot torso and the world that can directly command the base position or orientation.

So the robot may be highly actuated at the joint level, while still being underactuated at the system level.

This distinction matters.

For a floating-base humanoid, the generalized velocity contains both base motion and joint motion:

```cpp
v = [base_twist, joint_velocities]
```

If the robot has `na` actuated joints, then:

```cpp
nv = 6 + na
```

But the actuator torque vector only contains joint torques:

```cpp
tau.size() = na
```

The missing six dimensions are the floating-base degrees of freedom.

They are part of the dynamics, but they are not directly commanded.

This is why the actuation term has the structure:

```text
Sᵀ tau = [0_base_6, tau_actuated]
```

The base is controlled indirectly.

For a standing humanoid, the controller changes joint torques through the two leg kinematic chains. Those joint torques create ground reaction forces at the feet. Those contact forces then affect the motion of the floating base.

The control path is therefore:

```text
joint torques
→ leg kinematic chains
→ foot contact forces
→ whole-body dynamics
→ floating-base motion
```

So the humanoid is not underactuated because it lacks joint motors.

It is underactuated because the floating base is part of the generalized dynamics but has no direct actuator.

This is also why contacts are so important in humanoid WBC. Without support contacts, the floating base cannot be controlled in the same way. With support contacts, the robot can use the environment as a physical pathway to regulate base motion, center of mass behavior, and whole-body balance.

In code, this means we must keep these spaces separate:

```text
generalized dynamics space:
  base + joints

actuator space:
  joints only

contact force space:
  forces through the environment
```

Confusing these spaces leads to one of the most common mistakes in floating-base control: treating the humanoid as if it were just a fixed-base manipulator with more joints.

It is not.

The base is not fixed.
The base is not directly actuated.
The base is controlled through the body and the contacts.

## 3. Why nq, nv, and na must be separated

The first rule of floating-base robot code is to separate the meanings of `nq`, `nv`, and `na`.

```cpp
nq = configuration dimension
nv = velocity / acceleration dimension
na = actuator dimension
```

For fixed-base robots, these are often the same:

```cpp
nq = nv = na
```

For floating-base robots, they are usually not the same.

The main reason is orientation.

A free-flyer base is often stored using position plus quaternion:

```cpp
base configuration = position 3 + quaternion 4 = 7
```

But base velocity is represented as a 6D twist:

```cpp
base velocity = linear velocity 3 + angular velocity 3 = 6
```

So if the robot has 8 actuated joints:

```cpp
nq = 7 + 8 = 15
nv = 6 + 8 = 14
na = 8
```

The important consistency checks are:

```cpp
q.size()     == nq
v.size()     == nv
qddot.size() == nv
tau.size()   == na
```

This is one of the first places where floating-base code goes wrong.

In fixed-base code, it is common to write `qdot` and assume that it is the time derivative of `q` with the same dimension.

For floating-base robots, that assumption is dangerous.

The vector `v` is not simply a Euclidean derivative of the stored configuration vector `q`. In particular, quaternion orientation is stored using four numbers, but angular velocity is a 3D vector.

That is why dynamics libraries such as Pinocchio use the notation `v` for generalized velocity, rather than treating velocity as a raw `qdot` vector.

This distinction is not cosmetic.

It affects:

```text
state storage
numerical integration
pose error computation
Jacobian dimensions
mass matrix dimensions
solver variable dimensions
command conversion
```

If `q`, `v`, `qddot`, and `tau` are not clearly separated, the WBC implementation becomes fragile before the solver is even introduced.

---

## 4. Configuration is not always a Euclidean vector

A common source of confusion is the role of quaternions and Lie groups.

To store floating-base orientation, we need some representation. Common options include:

```text
Euler angles      3D, but singularity-prone
Quaternion        4D, with unit-norm constraint
Rotation matrix   9D, with orthogonality constraint
Lie algebra       3D, local representation
```

A question that often comes up is:

> If we use Lie group operations, do we still need quaternions?

Yes.

They solve different problems.

A quaternion is a storage representation for orientation. A Lie group operation is a way to compute differences, errors, and integration on the configuration manifold.

The roles are different:

```text
Storage:
  position + quaternion

Computation:
  integrate / difference / log on the Lie group

Velocity:
  tangent-space velocity
```

For storing the global orientation of a floating base, a quaternion is a practical choice. It avoids Euler-angle singularities and is more compact than a rotation matrix.

But a quaternion should not be treated like an ordinary 4D Euclidean vector during control.

For example, this is usually a bad idea:

```cpp
orientation_error = quat_des.coeffs() - quat.coeffs();
q_next = q + v * dt;
```

The first line computes an error in quaternion coefficient space, not a physically meaningful 3D orientation error.

The second line treats the configuration vector as if it lived in a flat Euclidean space. That is not true for floating-base configurations involving rotations.

Instead, pose differences and integration should be handled through manifold-aware operations.

Conceptually:

```cpp
v_error = difference(model, q_current, q_desired);
q_next  = integrate(model, q_current, v * dt);
```

The key idea is:

```text
q lives on the configuration manifold.
v lives in the tangent space.
```

For fixed-base robots with only revolute and prismatic joints, this distinction can be hidden most of the time. For floating-base robots, it becomes unavoidable.

## 5. The floating base makes contact part of actuation

A floating-base robot has more generalized velocity dimensions than actuator dimensions.

For a robot with `na` actuated joints:

```cpp
nv = 6 + na
```

but:

```cpp
tau.size() = na
```

The missing six dimensions correspond to the floating base. They are not commanded directly.

This means the WBC solver must operate in generalized dynamics space, while the hardware command exists only in actuator space.

For example, the solver may compute:

```cpp
qddot_sol  // size nv
lambda_sol // contact force dimension
tau_sol    // size na
```

The acceleration solution includes both base acceleration and joint acceleration. But only the joint torque part can be sent to actuators.

The base acceleration is not commanded directly. It is produced indirectly by the combination of actuator torques and contact forces.

This is why contact is not optional in floating-base WBC.

If the robot is standing, foot contact provides the physical path through which joint torques can influence the base. If the contact model is wrong, the base dynamics are wrong. If the contact force is infeasible, the desired motion is not physically realizable.

In code, this means the following spaces must remain separate:

```text
configuration space:
  q ∈ R^nq

generalized velocity / acceleration space:
  v, qddot ∈ R^nv

actuator space:
  tau ∈ R^na

contact force space:
  lambda ∈ R^nc
```

This separation should be visible in the API. If the code treats all of these as interchangeable vectors, the implementation will eventually fail in a hard-to-debug way.

---

## 6. Avoid exposing full generalized vector packing to users

One of the most dangerous API designs is to ask users to manually pack full generalized vectors.

For example:

```cpp
q.head<7>() = base_pose_as_xyz_quaternion;
q.tail(na) = joint_position;

v.head<6>() = base_twist;
v.tail(na) = joint_velocity;
```

This looks simple, but it creates many opportunities for mistakes.

```text
Is the base pose stored before or after the joints?
Is the quaternion ordering xyzw or wxyz?
Is the base velocity spatial or body-frame?
Is the base velocity 6D or 7D?
Is qdot the same thing as v?
Is tau dimension nq, nv, or na?
```

If every user of the API has to remember these conventions, the API is too easy to misuse.

A safer approach is to let users provide semantically meaningful state components, and let `RobotSystem` handle the generalized vector packing internally.

For example:

```cpp
struct FloatingBaseState {
  pinocchio::SE3 pose_world_base;
  pinocchio::Motion twist_world_base;
};
```

Then the public API can separate fixed-base and floating-base use cases:

```cpp
void updateState(ConstRefVector q_actuated,
                 ConstRefVector qdot_actuated,
                 ConstRefVector tau);

void updateState(ConstRefVector q_actuated,
                 ConstRefVector qdot_actuated,
                 ConstRefVector tau,
                 const FloatingBaseState& floating_base);
```

For a fixed-base robot:

```cpp
robot.updateState(q_actuated, qdot_actuated, tau);
```

For a floating-base robot:

```cpp
robot.updateState(q_actuated, qdot_actuated, tau, floating_base);
```

The user provides actuated joint state in the same way for both cases. The floating-base state is added only when the robot requires it.

Inside `RobotSystem`, the implementation can construct the full `q` and `v` according to the dynamics library convention.

Conceptually:

```cpp
// fixed-base
q = q_actuated;
v = qdot_actuated;

// floating-base
q = [base_position, base_quaternion, q_actuated];
v = [base_twist, qdot_actuated];
```

This is the important responsibility split:

```text
User:
  provides joint state and floating-base state semantically.

RobotSystem:
  converts semantic state into q and v conventions required by Pinocchio.

Controller:
  uses q, v, dynamics terms, tasks, and contacts to build the WBC problem.
```

The user should not need to memorize the generalized vector layout.

The code should enforce it.

## 7. Internal state should match the dynamics library

The public API should be semantic and hard to misuse.

Internally, however, it is often better for the canonical robot state to match the dynamics library.

For a Pinocchio-based WBC stack, an internal state may look like this:

```cpp
struct RobotState {
  Vector q;    // size nq
  Vector v;    // size nv
  Vector tau;  // size na
};
```

Here, `v` is generalized velocity.

For a fixed-base robot, `v` is essentially joint velocity.
For a floating-base robot, `v` is base twist plus joint velocity.

This makes the dynamics code clearer:

```cpp
M(q)        // nv x nv
h(q, v)     // nv
qddot       // nv
tau         // na
```

The WBC solver usually operates in `nv` space.

For example:

```cpp
decision variable = [qddot, lambda]

qddot ∈ R^nv
lambda ∈ R^contact_force_dim
```

The configuration vector `q` is a parameter of the dynamics problem. The acceleration variable `qddot` lives in tangent space and has dimension `nv`.

This is why the following is wrong for floating-base systems:

```cpp
qddot.size() == q.size()
```

The correct check is:

```cpp
qddot.size() == nv
```

A small dimension mistake here can propagate through the entire stack:

```text
wrong q/v assumption
→ wrong Jacobian dimension
→ wrong solver matrix
→ wrong command interpretation
→ unstable robot
```

In WBC code, dimension checks are not optional housekeeping. They are part of the safety structure.

## 8. Controllers should avoid raw quaternion arithmetic

Suppose we want to implement a base orientation task.

The task should compute an orientation error between the desired base orientation and the current base orientation.

A tempting implementation is:

```cpp
// Avoid
error = quat_des.coeffs() - quat.coeffs();
```

This should generally be avoided.

Quaternion coefficient difference is not the same thing as orientation error. It produces a 4D vector tied to the storage representation, not a 3D angular error suitable for a controller.

Instead, orientation error should be computed on the rotation group.

Conceptually:

```cpp
R_err = R_des.inverse() * R_current;
e_R   = log(R_err);
```

Depending on the convention, the order may be reversed. The important point is that the orientation error should be a 3D local angular error:

```cpp
Vector3 orientation_error;
```

The same idea applies to floating-base pose errors more generally.

The controller should not treat the pose as a raw vector. It should compute errors using the appropriate Lie group operations.

The practical rule is:

```text
Quaternion is for storage.
Lie group operations are for computation.
Controller errors live in tangent space.
```

This keeps task errors consistent with the velocity and acceleration spaces used by the solver.

## 9. Command variables are not the same as state variables

Another common source of confusion is the relationship between solver outputs and actuator commands.

An inverse-dynamics WBC solver may produce:

```cpp
qddot_sol
lambda_sol
tau_sol
```

But the hardware interface may expect a command structure such as:

```cpp
struct RobotCommand {
  Vector q;
  Vector qdot;
  Vector tau;
  Vector kp;
  Vector kd;
};
```

These command fields are not the same thing as the solver state variables.

For example, `qddot_sol` may need to be integrated to produce velocity and position commands:

```cpp
qdot_cmd = qdot_current + qddot_sol * dt;
q_cmd    = integrate(model, q_current, qdot_cmd * dt);
```

Then the command adapter may apply limits:

```text
velocity clamp
position clamp
torque clamp
rate limit
driver-specific scaling
```

This is why the solver layer and command layer should be separated.

The solver works in dynamics-level quantities:

```text
qddot_sol
lambda_sol
tau_sol
```

The command adapter produces hardware-friendly quantities:

```text
q_cmd
qdot_cmd
tau_ff_cmd
kp_cmd
kd_cmd
```

Mixing these layers creates ambiguity.

For example, if the solver output is named `q_cmd`, it becomes unclear whether this is:

```text
the desired task-space result
the optimization solution
the integrated joint command
the clamped hardware command
the actual value sent to the driver
```

A WBC stack should use naming that reflects the semantic layer.

A useful convention is:

```text
_des:
  desired value from planner, task, or reference generator

_sol:
  solver result

_cmd:
  command sent toward the hardware interface
```

The exact naming convention can vary, but the distinction should exist.

Solver results are not automatically actuator commands.

## 10. Recommended design rules

For floating-base WBC code, I use the following rules.

### Rule 1. Do not mix q, v, and tau dimensions

```cpp
q.size()     == nq
v.size()     == nv
qddot.size() == nv
tau.size()   == na
```

These should be checked explicitly.

### Rule 2. Do not expose generalized vector packing to users

Users should provide:

```cpp
q_actuated
qdot_actuated
tau
floating_base.pose
floating_base.twist
```

Full `q` and `v` packing should be owned by `RobotSystem`.

### Rule 3. Receive floating-base pose as SE3

```cpp
pinocchio::SE3 pose_world_base;
```

This avoids exposing quaternion ordering and raw pose-vector conventions at the public API level.

### Rule 4. Receive floating-base velocity as a twist

```cpp
pinocchio::Motion twist_world_base;
```

Base velocity is a 6D spatial motion, not a 7D quaternion derivative.

### Rule 5. Use Lie group operations for integration and difference

```cpp
q_next = integrate(model, q, v * dt);
dq     = difference(model, q0, q1);
```

Avoid raw vector addition and subtraction for floating-base configurations.

### Rule 6. Keep internal state aligned with the dynamics library

Inside `RobotSystem`, store canonical state as:

```cpp
q    // size nq
v    // size nv
tau  // size na
```

This makes dynamics, kinematics, and solver code consistent.

### Rule 7. Separate solver solution from hardware command

The WBC solver may produce:

```cpp
struct IDSolution {
  Vector qddot_sol;
  Vector lambda_sol;
  Vector tau_sol;
};
```

The command adapter may produce:

```cpp
struct RobotCommand {
  Vector q;
  Vector qdot;
  Vector tau;
  Vector kp;
  Vector kd;
};
```

The solver should not directly encode driver command policy.

## 11. Summary

Floating-base robot representation is the first hard problem in WBC software.

For fixed-base robots, it is often acceptable to think:

```cpp
q    = joint position
qdot = joint velocity
tau  = joint torque
```

For floating-base robots, this is not enough.

The configuration contains the base pose and actuated joint positions:

```cpp
q = [base pose, joint positions]
```

The generalized velocity contains base twist and actuated joint velocities:

```cpp
v = [base twist, joint velocities]
```

The actuator torque contains only actuated joint torques:

```cpp
tau = [joint torques]
```

Therefore:

```cpp
q.size()     == nq
v.size()     == nv
qddot.size() == nv
tau.size()   == na
```

For a floating-base robot with quaternion base orientation:

```cpp
nq = 7 + na
nv = 6 + na
```

and:

```cpp
tau.size() = na
```

This is the core reason floating-base robots are underactuated in the WBC formulation.

The mechanism may have actuators at nearly every joint, but the full dynamical system also includes the floating base. That base is not directly actuated. It is controlled indirectly through joint torques, kinematic chains, support contacts, and whole-body dynamics.

Quaternion and Lie group operations also serve different roles:

```text
Quaternion:
  storage representation for orientation

Lie group operations:
  integration, difference, and local error computation

Twist:
  tangent-space velocity representation
```

A good `RobotSystem` API should hide generalized vector packing from the user. The user should provide meaningful components: actuated joint state and floating-base state. Internally, `RobotSystem` can construct the `q` and `v` vectors required by the dynamics library.

This design keeps the controller from mixing configuration space, tangent space, actuator space, and contact force space.

That separation is not just a code style preference.

It is the foundation that makes the rest of WBC possible.

In the next post, we will move one layer up and look at the `RobotSystem` itself: what it should own, what it should expose, and how it should connect Pinocchio to the rest of the control architecture.
