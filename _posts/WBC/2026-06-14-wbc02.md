

제 2장 : Flaoting-Base Robot State

# Floating-Base Robot을 코드로 표현하는 법: q, v, Quaternion, 그리고 Lie Group

로봇 제어 코드를 짜다 보면 처음에는 모든 상태를 단순하게 생각하게 된다.

```cpp
q     // joint position
qdot  // joint velocity
tau   // joint torque
```

fixed-base manipulator나 robot hand만 다룰 때는 이 생각이 거의 맞다.
예를 들어 8-DoF robot hand라면 configuration, velocity, torque dimension이 모두 8이다.

```cpp
nq = 8
nv = 8
na = 8
```

그래서 `q`, `qdot`, `tau`를 같은 joint ordering으로 관리하면 된다.
하지만 floating-base robot이 들어오는 순간 이야기가 달라진다.

---

## 1. Fixed-base와 floating-base의 차이

Fixed-base system은 말 그대로 base가 world에 고정되어 있다.

예를 들어 테이블 위에 고정된 robot arm이나, wrist에 고정된 robot hand를 생각하면 된다.
이 경우 로봇의 움직임은 actuated joint들의 움직임만으로 표현된다.

```cpp
q = [joint positions]
v = [joint velocities]
```

반면 floating-base system은 base 자체가 움직인다.
Humanoid, quadruped, floating hand-arm system, free-floating robot 등이 여기에 해당한다.

이때 robot state에는 actuated joint뿐 아니라 base pose도 들어가야 한다.

```cpp
q = [base pose, joint positions]
v = [base velocity, joint velocities]
```

그런데 여기서 중요한 점이 있다.

Floating base는 6-DoF rigid body motion을 갖는다.

```text
translation: 3 DoF
rotation:    3 DoF
total:       6 DoF
```

그래서 floating-base robot은 fixed-base robot에 비해 물리적으로 6개의 generalized velocity가 추가된다.

하지만 이 6개의 base DoF에는 직접 torque를 걸 수 없다.
모터는 joint에만 달려 있고, floating base는 world에 고정되어 있지 않기 때문이다.

따라서 floating-base system은 일반적으로 underactuated system이다.

예를 들어 actuated joint가 8개인 floating-base robot이라면:

```cpp
nv = 6 + 8 = 14
na = 8
```

즉 dynamics equation은 14차원 공간에서 풀리지만, 실제 actuator torque command는 8개뿐이다.

이게 floating-base WBC에서 selection matrix가 필요한 이유다.

```text
M(q) qddot + h(q, v) = Sᵀ tau + Jᵀ lambda
```

여기서 `Sᵀ tau`는 actuated joint에만 torque를 넣고, floating base 쪽 6개 equation에는 직접 torque를 넣지 않는다.

개념적으로는 이런 구조다.

```text
Sᵀ tau = [0_base_6, tau_actuated]
```

## 2. nq, nv, na를 구분해야 하는 이유

Floating-base code에서 제일 먼저 명확히 해야 할 것은 `nq`, `nv`, `na`의 의미다.

```cpp
nq = configuration dimension
nv = velocity / acceleration dimension
na = actuator dimension
```

Fixed-base robot에서는 보통 이 셋이 같다.

```cpp
nq = nv = na
```

하지만 floating-base robot에서는 다를 수 있다.

대표적으로 free-flyer base를 quaternion으로 저장하면:

```cpp
base configuration = position 3 + quaternion 4 = 7
base velocity      = linear velocity 3 + angular velocity 3 = 6
```

따라서 actuated joint가 8개라면:

```cpp
nq = 7 + 8 = 15
nv = 6 + 8 = 14
na = 8
```

여기서 핵심은 이것이다.

```cpp
q.size()     == nq
v.size()     == nv
qddot.size() == nv
tau.size()   == na
```

`q`와 `v`의 dimension이 같지 않을 수 있다.
이게 floating-base code에서 `qdot`이라는 이름이 애매해지는 이유다.

Fixed-base에서는 `v == qdot`이라고 생각해도 거의 문제가 없다.
하지만 floating-base에서는 `v`가 단순한 `dq/dt` vector가 아니다.
특히 quaternion orientation은 4개의 숫자로 저장되지만, 그 회전 속도는 3D angular velocity로 표현된다.

그래서 Pinocchio 같은 dynamics library에서는 velocity를 `qdot`이 아니라 `v`라고 부른다.

---

## 3. Quaternion은 저장 형식이고, Lie Group은 계산 방식이다

Floating-base pose를 저장하려면 orientation을 어떤 형식으로 표현해야 한다.

대표적인 선택지는 다음과 같다.

```text
Euler angle      3D, but singularity
Quaternion       4D, unit constraint
Rotation matrix  9D, orthogonality constraint
Lie algebra      3D, local representation
```

제어 코드를 짤 때 자주 헷갈리는 부분은 이것이다.

> Lie Group으로 계산하면 quaternion이 필요 없는 것 아닌가?

그렇지 않다.

Lie Group은 pose error, integration, difference를 계산하는 방식이고, quaternion은 orientation을 메모리에 저장하는 방식이다.

즉 역할이 다르다.

```text
Storage:      position + quaternion
Computation:  Lie group integrate / difference / log
Velocity:     tangent-space velocity
```

로봇의 현재 base orientation을 전역적으로 저장하려면 quaternion이 좋은 선택이다.
Euler angle처럼 singularity가 없고, rotation matrix보다 compact하다.

하지만 quaternion을 직접 빼거나 원소별로 적분하면 안 된다.

나쁜 예시는 이런 것이다.

```cpp
orientation_error = quat_des.coeffs() - quat.coeffs();
q_next = q + v * dt;
```

이 방식은 floating-base pose가 일반적인 Euclidean vector가 아니라는 사실을 무시한다.

대신 pose error와 integration은 Lie Group operation으로 처리해야 한다.

개념적으로는 다음과 같다.

```cpp
v_error = difference(model, q_current, q_desired);
q_next  = integrate(model, q_current, v * dt);
```

즉 configuration은 manifold 위에 있고, velocity는 그 configuration의 tangent space에 있다.

## 4. RobotSystem API는 사용자가 generalized vector를 직접 만들지 않게 해야 한다

Floating-base code에서 가장 위험한 API는 사용자에게 full generalized vector를 직접 만들게 하는 것이다.

예를 들어 이런 코드는 사고가 나기 쉽다.

```cpp
q.head<7>() = base_pose_as_xyz_quaternion;
q.tail(na) = joint_position;

v.head<6>() = base_twist;
v.tail(na) = joint_velocity;
```

여기에는 여러 가지 함정이 있다.

```text
base pose가 앞인가 뒤인가?
quaternion ordering이 xyzw인가 wxyz인가?
base velocity는 6D인가 7D인가?
qdot과 v는 같은 것인가?
joint torque dimension은 nq인가 nv인가 na인가?
```

사용자가 매번 이걸 기억해야 한다면 API 설계가 너무 위험하다.

더 나은 방식은 joint state와 floating-base state를 분리해서 받는 것이다.

예를 들어:

```cpp
struct FloatingBaseState {
  pinocchio::SE3 pose_world_base;
  pinocchio::Motion twist_world_base;
};
```

그리고 `RobotSystem`의 public API는 이렇게 만들 수 있다.

```cpp
void updateState(ConstRefVector q_actuated,
                 ConstRefVector qdot_actuated,
                 ConstRefVector tau);

void updateState(ConstRefVector q_actuated,
                 ConstRefVector qdot_actuated,
                 ConstRefVector tau,
                 const FloatingBaseState& floating_base);
```

사용자는 fixed-base든 floating-base든 actuated joint state를 같은 방식으로 넣는다.

Fixed-base robot에서는:

```cpp
robot.updateState(q_actuated, qdot_actuated, tau);
```

Floating-base robot에서는:

```cpp
robot.updateState(q_actuated, qdot_actuated, tau, floating_base);
```

이렇게 하면 사용자는 generalized vector packing을 외울 필요가 없다.
`RobotSystem` 내부에서만 Pinocchio convention에 맞게 `q`와 `v`를 만든다.

개념적으로 내부에서는 다음과 같은 일이 일어난다.

```cpp
// fixed-base
q = q_actuated;
v = qdot_actuated;

// floating-base
q = [base_position, base_quaternion, q_actuated];
v = [base_twist, qdot_actuated];
```

이 설계의 핵심은 responsibility를 나누는 것이다.

```text
User:
  joint state와 floating base state를 의미적으로 제공한다.

RobotSystem:
  Pinocchio가 기대하는 q, v vector로 변환한다.

Controller:
  q, v, dynamics terms를 사용해 WBC problem을 만든다.
```

## 5. 내부 state는 q, v, tau로 두는 것이 자연스럽다

Public API에서는 `q_actuated`, `qdot_actuated`라는 이름이 사용자에게 친절하다.

하지만 `RobotSystem` 내부 canonical state는 Pinocchio와 맞추는 것이 좋다.

```cpp
struct RobotState {
  Vector q;    // size nq
  Vector v;    // size nv
  Vector tau;  // size na
};
```

여기서 `v`는 generalized velocity다.
Fixed-base에서는 사실상 joint velocity이고, floating-base에서는 base twist와 joint velocity를 합친 tangent-space velocity다.

이렇게 해두면 dynamics code가 명확해진다.

```cpp
M(q)        // nv x nv
h(q, v)     // nv
qddot       // nv
tau         // na
```

WBC solver도 보통 `nv` 공간에서 문제를 푼다.

```cpp
decision variable = [qddot, lambda]
qddot ∈ R^nv
lambda ∈ contact force dimension
```

`q`는 configuration parameter이고, `qddot`은 tangent-space acceleration이다.
Floating-base에서 `qddot`의 dimension은 `nq`가 아니라 `nv`다.

## 6. Controller에서는 raw quaternion을 거의 만지지 않는 것이 좋다

Base orientation task를 만든다고 하자.

이때 orientation error를 quaternion coefficient difference로 계산하면 안 된다.

```cpp
// Avoid
error = quat_des.coeffs() - quat.coeffs();
```

대신 rotation error는 Lie Group의 local error로 계산하는 것이 자연스럽다.

개념적으로는:

```cpp
R_err = R_des.inverse() * R_current;
e_R   = log(R_err);
```

혹은 convention에 따라 반대 방향을 사용할 수도 있다.

중요한 것은 orientation error가 4D quaternion difference가 아니라 3D local angular error라는 점이다.

```cpp
Vector3 orientation_error;
```

즉 quaternion은 저장 표현이고, controller error는 tangent space에서 계산한다.

## 7. 추천하는 코드 설계 원칙

Floating-base WBC code를 작성할 때 나는 다음 규칙을 추천한다.

### Rule 1. q, v, tau dimension을 절대 섞지 않는다

```cpp
q.size()     == nq
v.size()     == nv
qddot.size() == nv
tau.size()   == na
```

### Rule 2. 사용자는 full generalized vector를 직접 pack하지 않는다

사용자는 다음만 제공한다.

```cpp
q_actuated
qdot_actuated
tau
floating_base.pose
floating_base.twist
```

Full `q`, `v` packing은 `RobotSystem`이 담당한다.

### Rule 3. Floating-base pose는 SE3로 받는다

```cpp
pinocchio::SE3 pose_world_base;
```

사용자가 quaternion ordering을 직접 다루지 않게 한다.

### Rule 4. Floating-base velocity는 twist로 받는다

```cpp
pinocchio::Motion twist_world_base;
```

Base velocity는 7D quaternion derivative가 아니라 6D spatial velocity다.

### Rule 5. Integration과 difference는 Lie Group API를 사용한다

```cpp
q_next = integrate(model, q, v * dt);
dq     = difference(model, q0, q1);
```

Raw vector addition/subtraction을 피한다.

### Rule 6. Solver와 command layer를 분리한다

WBC solver는 `qddot`, `lambda`, `tau`를 계산한다.
Command adapter는 이 결과를 hardware-friendly command로 변환한다.

```cpp
IDSolution:
  qddot_sol
  lambda_sol
  tau_cmd

CommandAdapter:
  q_cmd
  qdot_cmd
  kp
  kd
  tau_ff
```

이렇게 해야 solver semantic과 actuator command policy가 섞이지 않는다.

---

## 결론

Floating-base robot을 코드로 표현할 때 가장 중요한 것은 `q`, `v`, `tau`의 의미를 분리하는 것이다.

Fixed-base robot에서는 모든 것이 단순하다.

```cpp
q = joint position
v = joint velocity
tau = joint torque
```

하지만 floating-base robot에서는 base pose가 configuration에 추가되고, base velocity는 tangent-space twist로 표현된다.

```cpp
q = [base pose 7D, joint positions]
v = [base twist 6D, joint velocities]
tau = [joint torques]
```

따라서 `nq`, `nv`, `na`는 서로 다를 수 있다.

Quaternion은 base orientation을 저장하기 위한 표현이다.
Lie Group은 pose difference, integration, orientation error를 계산하기 위한 방법이다.

둘은 경쟁 관계가 아니다.

```text
Quaternion: storage
Lie Group: computation
Twist: velocity
```

좋은 floating-base API는 사용자가 generalized vector packing을 외우게 하지 않는다.
대신 joint state와 floating-base state를 의미적으로 받으며, 내부에서 Pinocchio convention에 맞게 `q`와 `v`를 구성한다.

이렇게 하면 fixed-base hand부터 floating-base humanoid까지 같은 WBC core 위에서 다룰 수 있고, 동시에 quaternion, tangent velocity, actuator dimension이 뒤섞이는 실수를 줄일 수 있다.
